# ctx-link

**Connect Claude Code sessions across projects via shared context bundles.**

When you're building a feature that spans multiple repos (e.g., backend API + frontend), each Claude Code session only knows about its own project. `ctx-link` lets sessions share context through bundles — so Claude in the frontend repo knows what changed in the backend, and vice versa.

---

## How it works

```
Claude Code (Project A)
    │
    │ git commit
    ▼
PostToolUse hook detects commit
    │
    │ ctx-link push (async, non-blocking)
    ▼
Summary stored in bundle
    │
    │  local mode → ~/.ctx-link/local/<id>/entries.json
    │  cloud mode → Supabase (shared across machines)
    ▼
Claude Code (Project B)
    │
    │ ctx-link pull (or ask Claude: "pull context")
    ▼
Claude sees what Project A did
```

**Key design decisions:**
- Summaries are generated by Claude Code itself (the AI calling the MCP tool), not a separate API call. No Anthropic API key needed.
- Three modes: **off** (default), **local** (same machine), **cloud** (cross-machine via Supabase).
- Cloud bundles are team-scoped. Team members authenticate with name + password.

## Quick start

### Install

```bash
git clone https://github.com/HassanKourani/Context-Linker.git ctx-link
cd ctx-link
bun install
```

### Make CLI globally accessible

```bash
cd packages/cli && bun link
echo 'export PATH="$HOME/.bun/bin:$PATH"' >> ~/.zshrc && source ~/.zshrc
```

### Register MCP server with Claude Code

```bash
claude mcp add --scope user ctx-link -- bun /absolute/path/to/ctx-link/packages/mcp-server/src/index.ts
```

Restart Claude Code. Run `/mcp` to verify `ctx-link` is connected.

### (Optional) Auto-push hook

Add to `~/.claude/settings.json` in the `PostToolUse` array:

```json
{
  "matcher": "Bash",
  "hooks": [{
    "type": "command",
    "command": "/absolute/path/to/ctx-link/packages/hooks/claude-code-hook.sh"
  }]
}
```

This auto-pushes context when Claude Code runs `git commit` or `gh pr create`. Only fires in projects with mode `"local"` or `"cloud"` — projects with `"off"` (or no `.ctx-link.json`) are ignored.

## Three modes

### Mode: off (default)

Every project starts in this mode. No linking, no data stored, hooks do nothing.

### Mode: local

For FE + BE on the same machine. Entries stored in `~/.ctx-link/local/<bundle_id>/entries.json`. No network, no tokens, no Supabase.

```bash
# In your frontend repo:
ctx-link create my-feature --mode local
# outputs bundle ID

# In your backend repo:
ctx-link join <bundle_id> <token> --mode local
```

### Mode: cloud

For cross-machine or team use. Entries stored in Supabase. Requires team membership.

```bash
# Create a team (once)
ctx-link team-create my-team --password "secret"

# Teammate joins the team
ctx-link team-join my-team --password "secret"

# Create a bundle under the team
ctx-link team-list  # grab the team ID
ctx-link create my-feature --mode cloud --team <team_id>

# Teammate joins the bundle (no token needed — team membership is the auth)
ctx-link join <bundle_id> --mode cloud
```

## CLI reference

### Teams

| Command | Description |
|---------|-------------|
| `ctx-link team-create <name> --password <pw>` | Create a new team. Creator auto-joins. |
| `ctx-link team-join <name> --password <pw>` | Join an existing team by name + password. |
| `ctx-link team-list` | List teams you belong to. |

### Bundles

| Command | Description |
|---------|-------------|
| `ctx-link create <name> --mode local` | Create a local bundle (same machine). |
| `ctx-link create <name> --mode cloud --team <id>` | Create a cloud bundle under a team. |
| `ctx-link join <bundle_id> [token] --mode local` | Join a local bundle. |
| `ctx-link join <bundle_id> --mode cloud` | Join a cloud bundle (must be in the team). |
| `ctx-link list` | List bundles this machine has joined. |
| `ctx-link status <bundle_id>` | Show session count, entry count, last activity. |
| `ctx-link delete-bundle <bundle_id>` | Permanently delete a bundle and all its entries. |

### Push / Pull

| Command | Description |
|---------|-------------|
| `ctx-link push --message <text>` | Push a summary to all project bundles. |
| `ctx-link push --diff` | Push using git diff; auto-extracts commit message as summary. |
| `ctx-link push --diff --summary <text>` | Push git diff with explicit summary. |
| `ctx-link push --event <type> --ref <ref>` | Specify event type and reference (commit SHA, PR#). |
| `ctx-link pull` | Pull entries from all project bundles (excludes own project). |
| `ctx-link pull <bundle_id>` | Pull from one specific bundle. |
| `ctx-link pull --include-self` | Include your own project's entries. |
| `ctx-link pull --since <iso> --limit <n>` | Filter by time and count. |

**Event types:** `commit`, `pr_open`, `manual`, `session_end`

### Rewind / Restore

| Command | Description |
|---------|-------------|
| `ctx-link rewind --bundle <id> --project <name> --last-n <n>` | Dry-run: preview which entries would be rewound. |
| `ctx-link rewind ... --apply` | Actually apply the rewind. |
| `ctx-link rewind ... --apply --reason "text"` | Apply with an audit reason. |
| `ctx-link rewind ... --since <iso>` | Rewind entries since a timestamp. |
| `ctx-link rewind ... --entry-ids <a,b,c>` | Rewind specific entries. |
| `ctx-link rewind ... --after-ref <sha>` | Rewind everything after a commit (keeps the pivot). |
| `ctx-link rewind ... --force` | Override the 50-entry safety cap. |
| `ctx-link restore --bundle <id> --project <name>` | Restore all rewound entries for a project. |
| `ctx-link restore ... --from-log <log_id>` | Restore entries from a specific rewind operation. |
| `ctx-link restore ... --entry-ids <a,b,c>` | Restore specific entries. |
| `ctx-link rewind-history <bundle_id>` | List past rewinds. |
| `ctx-link rewind-history <bundle_id> --project <name>` | Filter by project. |

## MCP tool reference

These are the tools Claude Code sees when the MCP server is connected.

| Tool | Description |
|------|-------------|
| `bundle_create` | Create a new bundle (requires mode + team for cloud). |
| `bundle_join` | Join a bundle (cloud: checks team membership). |
| `bundle_list` | List bundles this machine has joined. |
| `bundle_status` | Get session count, entry count, last activity. |
| `bundle_delete` | Permanently delete a bundle (irreversible). |
| `context_push` | Push a context entry. Claude generates the summary in the `summary` field. |
| `context_pull` | Pull recent entries, formatted for Claude's context. |
| `context_rewind` | Soft-delete entries scoped to one project. |
| `context_restore` | Undo a rewind. |
| `rewind_history` | List past rewinds. |

**Usage from Claude Code** — just ask naturally:
- "Pull context from my bundle"
- "Push what I just did to the bundle"
- "Check bundle status"
- "Rewind the last 3 entries from this project"

## Architecture

```
ctx-link/
  packages/
    core/          # Shared logic: config, Supabase, bundles, entries, teams,
    |              # rewind, local-store — no transport layer
    mcp-server/    # MCP protocol wrapper (stdio) with 10 tools
    cli/           # ctx-link CLI (commander) for manual ops + hook targets
    hooks/         # Git post-commit + Claude Code PostToolUse hook scripts
  supabase/
    migrations/    # SQL schema (teams, bundles, sessions, entries, rewind_log)
```

### Data model

```
teams            -- access control container
  id              uuid
  name            text (unique)
  password_hash   text (argon2)
  created_at      timestamptz

team_members     -- who belongs to which team
  id              uuid
  team_id         uuid → teams
  machine_id      text
  joined_at       timestamptz

bundles          -- shared context container, belongs to a team
  id              uuid
  name            text
  team_id         uuid → teams
  created_at      timestamptz
  created_by      text

sessions         -- one per (project, machine) in a bundle
  id              uuid
  bundle_id       uuid → bundles
  project_name    text
  machine_id      text

entries          -- the context handoff notes
  id              uuid
  bundle_id       uuid → bundles
  session_id      uuid → sessions
  event_type      text (commit | pr_open | manual | session_end)
  trigger_ref     text (commit SHA, PR number)
  summary         text
  files_touched   text[]
  decisions       jsonb
  superseded_at   timestamptz (soft-delete for rewind)

rewind_log       -- audit trail
  id, bundle_id, project_name, strategy_kind, strategy_detail,
  affected_entry_ids, affected_count, reason, performed_by, performed_at
```

### Local mode storage

```
~/.ctx-link/
  config.json        # machine_id (auto-generated)
  tokens.json        # bundle tokens (mode 600)
  teams.json         # team memberships (mode 600)
  local/
    <bundle_id>/
      meta.json      # bundle name, created_at
      entries.json   # array of entries
```

## Security model

- **Cloud bundles are team-gated.** You must join the team (name + password) before accessing any bundle in it.
- **Team passwords are hashed.** Supabase only stores argon2 hashes.
- **Local bundles have no auth.** They're files on your machine — if you can read the filesystem, you can read the entries.
- **Bundle IDs are UUIDs.** Not guessable; no "list all bundles" endpoint.
- **Summaries are plaintext in Supabase.** E2E encryption is a future improvement.

## Troubleshooting

### "No bundles configured for this project"
Run `ctx-link create` or `ctx-link join` in this directory first.

### "You are not a member of this team"
Run `ctx-link team-join <name> --password <pw>` to join the team first.

### "Cloud bundles require --team"
Run `ctx-link team-list` to find your team ID, then pass `--team <id>`.

### Claude Code doesn't see ctx-link MCP server
- Run `/mcp` in Claude Code to check status.
- Verify you ran `claude mcp add --scope user ctx-link -- bun /path/to/mcp-server/src/index.ts`.
- Check `bun` is in your PATH.

### Hooks aren't firing
- Check `~/.claude/settings.json` has the PostToolUse hook entry.
- Verify the hook script path is absolute and executable (`chmod +x`).
- Hooks only fire for projects with `mode: "local"` or `"cloud"` in `.ctx-link.json`.

### Rewind affected too many entries
The safety cap (50) refuses bulk rewinds without `--force`.
